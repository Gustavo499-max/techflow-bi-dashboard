Diagnóstico: IA amplificando caos, não valor
O volume de code commits cresceu 47% após adoção de IA, porém a Change Failure Rate subiu de 9% para 18,4%. O Lead Time permanece em 4,2 dias — indicando que o gargalo está no code review, não na escrita de código. A IA está acelerando a produção de código, mas os processos de qualidade e revisão não escalaram no mesmo ritmo.
🎯
Oportunidade identificada
Squads que combinam IA + processos estruturados de review apresentam Lead Time 62% menor e CFR abaixo de 5%. A solução não é remover a IA, mas instrumentar o pipeline com BI para identificar onde a qualidade se perde e atuar preventivamente.

-- ============================================================
-- DIMENSÃO: dim_time
-- ============================================================
CREATE TABLE dim_time (
  time_key INTEGER PRIMARY KEY,
  data_deploy DATE NOT NULL,
  ano SMALLINT,
  semestre SMALLINT,
  trimestre SMALLINT,
  mes SMALLINT,
  semana_ano SMALLINT,
  dia_semana VARCHAR(15),
  hora_deploy SMALLINT,
  e_horario_comercial BOOLEAN
);

-- ============================================================
-- DIMENSÃO: dim_repository
-- ============================================================
CREATE TABLE dim_repository (
  repo_key INTEGER PRIMARY KEY,
  github_repo_id BIGINT UNIQUE NOT NULL,
  nome_repo VARCHAR(255),
  squad_owner VARCHAR(100),
  linguagem_principal VARCHAR(50),
  tipo VARCHAR(30), -- backend | frontend | mobile | infra
  criticidade VARCHAR(10) -- P0 | P1 | P2
);

-- ============================================================
-- DIMENSÃO: dim_engineer
-- ============================================================
CREATE TABLE dim_engineer (
  eng_key INTEGER PRIMARY KEY,
  github_login VARCHAR(100) UNIQUE,
  nome_engenheiro VARCHAR(200),
  squad VARCHAR(100),
  nivel VARCHAR(10), -- jr | pl | sr | staff
  usa_ia_assistida BOOLEAN,
  ferramenta_ia VARCHAR(50), -- copilot | cursor | codewhisperer
  anos_experiencia SMALLINT
);

-- ============================================================
-- DIMENSÃO: dim_feature
-- ============================================================
CREATE TABLE dim_feature (
  feature_key INTEGER PRIMARY KEY,
  jira_ticket_id VARCHAR(30) UNIQUE NOT NULL,
  tipo VARCHAR(20), -- bug | feature | chore | refactor
  epico VARCHAR(255),
  squad VARCHAR(100),
  prioridade VARCHAR(10) -- Critical | High | Medium | Low
);

-- ============================================================
-- FATO: fato_deploys (tabela central)
-- ============================================================
CREATE TABLE fato_deploys (
  deploy_id BIGINT PRIMARY KEY,
  time_key INTEGER REFERENCES dim_time,
  repo_key INTEGER REFERENCES dim_repository,
  eng_key INTEGER REFERENCES dim_engineer,
  feature_key INTEGER REFERENCES dim_feature,
  -- Métricas de velocidade
  lead_time_horas FLOAT,
  tempo_review_horas FLOAT,
  qtd_revisores INTEGER,
  -- Métricas de volume
  qtd_commits INTEGER,
  linhas_adicionadas INTEGER,
  linhas_removidas INTEGER,
  codigo_ia_pct FLOAT, -- 0.0 a 1.0
  -- Métricas de qualidade
  deploy_sucesso BOOLEAN,
  falhou_producao BOOLEAN,
  mttr_minutos FLOAT,
  -- Métricas de valor de produto
  conversao_produto_delta FLOAT -- variação % conversão 7d pós-deploy
);
SQL de Análise — Consultas Estratégicas
Queries para cálculo das métricas DORA
-- Lead Time médio por squad (últimos 90 dias)
SELECT
  r.squad_owner,
  ROUND(AVG(f.lead_time_horas) / 24, 1) AS lead_time_dias,
  ROUND(AVG(f.tempo_review_horas), 1) AS avg_review_horas,
  COUNT(*) AS total_deploys,
  ROUND(SUM(f.falhou_producao::INT) * 100.0 / COUNT(*), 1) AS cfr_pct
FROM fato_deploys f
JOIN dim_repository r ON f.repo_key = r.repo_key
JOIN dim_time t ON f.time_key = t.time_key
WHERE t.data_deploy >= CURRENT_DATE - INTERVAL '90 days'
GROUP BY r.squad_owner
ORDER BY lead_time_dias ASC;

-- Impacto da IA na qualidade do código
SELECT
  CASE
    WHEN f.codigo_ia_pct < 0.3 THEN 'baixo_ia'
    WHEN f.codigo_ia_pct < 0.7 THEN 'medio_ia'
    ELSE 'alto_ia'
  END AS faixa_ia,
  COUNT(*) AS total_deploys,
  ROUND(AVG(f.lead_time_horas),1) AS avg_lead_time_h,
  ROUND(AVG(f.falhou_producao::INT)*100,1) AS cfr_pct,
  ROUND(AVG(f.conversao_produto_delta)*100,2) AS impacto_conversao_pct
FROM fato_deploys f
GROUP BY faixa_ia
ORDER BY faixa_ia;

Por que ELT e não ETL?
Justificativa técnica da arquitetura
❌ ETL — Problemas neste contexto
→ Transformação antes da carga = pipeline frágil
→ Reprocessamento caro se lógica mudar
→ Difícil exploração ad-hoc dos dados brutos
→ Ferramentas clássicas (Informatica, SSIS) não escalam bem com APIs REST em tempo real
→ Data lineage difícil de auditar
✅ ELT — Vantagens para a TechFlow
→ Dados brutos preservados no Data Lake (auditoria total)
→ BigQuery processa SQL em escala petabyte nativamente
→ dbt permite iterar na transformação sem re-extrair
→ Novas métricas = novo model dbt, não novo pipeline
→ Time de engenharia já conhece SQL (curva de aprendizado baixa)

Recomendações Estratégicas para o CTO
🔬
1. Instrumentar o Code Review como prioridade imediata
O gargalo principal está no review (65% do lead time). Implementar revisão assistida por IA (ex: CodeRabbit, GitHub Copilot Review) pode reduzir o tempo de review de 4,8h para 1,5h. Adicionalmente, definir SLA de review por prioridade: features P0 = máximo 2h, features P1 = máximo 8h.
📏
2. Definir política de PR size limit
PRs com mais de 400 linhas têm 3,2x maior probabilidade de falha e 5,1x maior tempo de review. Implementar hard limit de 500 linhas por PR e quebrar features complexas em múltiplos PRs atômicos. O BI monitorará esse KPI semanalmente via fato_deploys.linhas_adicionadas.
🤖
3. Criar "AI Usage Guidelines" com gates de qualidade
Engenheiros com alto uso de IA (>70%) sem processo estruturado apresentam CFR de 31%. Implementar checklist obrigatório pré-merge para código gerado por IA: testes unitários mínimos 80% coverage, revisão de 2+ engenheiros sênior, análise estática (SonarQube) passando. A coluna fato_deploys.codigo_ia_pct + deploy_sucesso permitirá medir o impacto desta política.
📈
4. Conectar metas de engenharia a OKRs de produto
Os dados mostram correlação positiva (r=0,72) entre deploy frequency e taxa de conversão do produto. Squads com 7+ deploys/semana apresentaram +4,2% de conversão no período analisado. Proposta: incluir métricas DORA como KR no OKR Q3 da engenharia, com dashboard de progresso acessível ao CTO em tempo real.

IA aumentando volume de código, mas amplificando instabilidade. CFR 18,4% — 3,7x acima do nível elite. Code review é o gargalo crítico.
